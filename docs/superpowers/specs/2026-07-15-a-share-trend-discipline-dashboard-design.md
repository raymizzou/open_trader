# A 股趋势纪律与 Dashboard 报告重设计

## 状态

- 日期：2026-07-15
- 状态：待用户审阅
- 范围：仅东方财富 A 股趋势报告

## 目标

把参考看板中的 A 股买卖纪律变成真实报告规则，并将 Dashboard 报告改成
与现有持仓主页面一致的动作优先看板。规则上线后，用新规则覆盖并重生成
`2026-07-15` A 股报告；不修改美股或港股趋势报告，不补发飞书。

## 已选方案

采用“动作优先”的单券商报告：用户从东方财富账户区点击 `当天趋势报告`，
进入独立 A 股报告工作区。首屏依次展示卖出、买入和持有动作；买卖纪律放在
动作之后，完整候选审计默认折叠。

视觉层完全复用现有 Dashboard：系统字体、白色卡片、蓝灰边框、蓝色主操作、
现有圆角与阴影。红色和绿色只作为卖出和买入的辅助语义，始终同时显示文字。
不引入米色主题、字体、图标库或新的设计系统。

## 买入纪律

候选必须通过以下 12 条纪律：

1. 资产必须是 `A股`。
2. 前一数据日趋势温度为 `温`，当前温度为 `热` 或 `沸`。
3. 当前温度为 `热` 时目标仓位为账户净值的 4%；为 `沸` 时为 2%。
4. Trend Animals `priceIndex` 小于或等于 200 元。
5. 本地趋势强度大于或等于 95。
6. 排除 ETF 和其他基金。
7. 所属行业当前温度必须高于 `温`，即只能为 `热` 或 `沸`。
8. 趋势节气不得晚于 `夏至`；允许 `谷雨`、`立夏` 和 `夏至`。
9. Trend Animals 市值大于或等于 100 亿元。
10. 排除名称包含 ST 或退市标识的标的。
11. 日成交额大于或等于 2 亿元。
12. 排除北交所标的。

现有 `右侧天数 < 10` 门槛被趋势节气规则替代，不再同时执行。趋势节气按
Trend Animals 的右侧生命周期含义使用：`谷雨` 为右侧震荡，`立夏` 为萌芽，
`夏至` 为发展，`小暑` 为破圈加速，`大暑` 为爆发，`立秋` 为右侧结束。

以下现有安全门槛继续执行，但不计入参考看板的 12 条策略纪律：

- 当前可交易。
- 数据日期与报告数据日一致。
- 当前账户尚未持有。
- 处于右侧趋势。
- 危险信号明确为否。
- 名称、资产、交易所等身份字段完整且合法。
- ATR14 与执行参考收盘价可用。

任何必需字段缺失都不能进入买入名单；候选进入排除项并记录具体缺失原因，
不得转为可人工放行的买入。

## 候选排序与正式买入

通过纪律的候选沿用确定性排序：

1. 趋势强度降序。
2. 右侧天数升序。
3. 日成交额降序。
4. 证券代码升序。

主表只显示同时满足策略纪律、账户现金和持仓席位约束的正式买入计划。组合
最多持有 10 个标的；A 股数量按 100 股向下取整；现金不足一手时跳过该候选，
且不消耗席位。其他合格候选保留在审计详情，不在主表冒充可执行动作。

账户净值和现金使用最近一次有效账户快照。账户快照有效但非实时，不取消策略
动作；报告显示 `账户数据非实时，执行前核对现金与持仓`。账户快照缺失或结构
无效仍使报告生成失败。

## 两个价格源

两个价格必须同时保留，不能合并成含义模糊的“价格”：

- `筛选价`：Trend Animals `priceIndex`，只用于 `≤ 200 元` 纪律。
- `执行参考价`：富途数据日的前复权日线收盘价，用于目标股数、ATR14 和预计
  初始保护线。

主表同时展示并标注两个价格来源；审计 JSON 保存两者。预计初始保护线继续为：

```text
执行参考价 - 2 × ATR14
```

## 行业温度数据流

候选快照新增读取：

- `industryTmId`
- `priceIndex`
- `marketCap`
- `trendTemperatureCurr`
- `trendTemperaturePrev`
- `trendPhaseCurr`

从候选快照收集去重后的 `industryTmId`，再批量请求一次行业快照，只读取行业
当前温度。禁止逐候选请求行业数据。报告保存候选的行业 ID、行业名称和行业
温度，使 Dashboard 只负责展示，不重新调用 API 或计算规则。

## 卖出与保护纪律

持仓动作按以下优先级判断：

1. 活动保护线已经触发：全部卖出。
2. 危险信号触发：全部卖出。
3. 离开右侧趋势：全部卖出。
4. 前一温度为 `温`、`热` 或 `沸`，当前变为 `平`：全部卖出。
5. 否则继续持有。

温度已经连续为 `平` 不构成新的“温度转平”事件；离开右侧规则仍可独立触发
卖出。判断卖出所需字段缺失时沿用人工复核，不静默继续持有。

`沸腾` 或 `开香槟` 不减仓，只启动或继续上移活动保护线：

```text
新保护线 = max(旧保护线, 前 5 个交易日最低价)
```

保护线不得下移。Trend Animals 当前没有“波动率放大”字段，本设计不实现该
分支，也不显示虚假的“减仓 30%”。

## 已有持仓提示

买入纪律只约束新开仓，不因已有持仓不再满足价格、市值、强度、行业温度或
节气门槛而自动卖出。

当相应数据存在时，持仓行只显示当前未满足的买入条件，例如：

- `强度 91.3，低于入场线 95`
- `节气已到小暑`
- `不是新的温转热入场信号`

通过项不展示。缺少可选提示数据时显示 `数据不可用`，但只有卖出判断所需字段
缺失才把动作升级为人工复核。

## Dashboard 桌面布局

报告工作区按以下顺序展示：

1. 券商、市场、报告日期、数据日期、生成时间和账户状态。
2. 正式买入、全部卖出、继续持有、人工复核四个计数卡。
3. `优先处理 · 卖出触发` 表。
4. `09:30–10:00 · 正式买入计划` 表。
5. `盘中持续 · 已有持仓` 表。
6. 买入纪律和卖出纪律卡片。
7. 默认折叠的审计详情。

桌面买入表至少显示：标的、筛选价、执行参考价、温度变化、节气、强度、行业
温度、市值、日成交额、目标仓位、目标金额、预计数量和预计保护线。

卖出表至少显示：标的、执行参考价、温度变化、强度、触发原因、活动保护线、
持仓提示和动作。

## Dashboard 移动布局

在 760px 以下，动作表转换为纵向事实卡，不保留横向滚动表。每张卡先显示标的
和动作，再显示两列事实网格。所有交互目标至少 44px 高。

移动端顺序与桌面一致，但买入纪律、卖出纪律和审计详情默认折叠。页面在 375px
视口不得产生横向滚动；颜色不得成为识别动作的唯一方式。

## 审计详情

审计详情保存并展示：

- 所有原始候选及最终排名。
- 每个候选逐项排除原因。
- 个股温度、节气、行业 ID 和行业温度。
- Trend Animals 筛选价与富途执行参考价。
- 市值、日成交额、ATR14 和危险信号。
- 数据来源、API 付费字段、实际或估算成本。

主表不展示完整通过/失败矩阵；审计 JSON 必须保留足够事实重建每个排除结论。

## 一次性覆盖重生成

新规则部署后，对 `reports/trend_a_share/2026-07-15.json` 和同名 Markdown 执行
一次性覆盖重生成：

- 不创建 `-r1` revision。
- 不保留旧报告副本。
- 新 JSON 与 Markdown 必须先完整生成，再原子替换旧文件，不能留下半对产物。
- 使用新规则重新读取真实 Trend Animals、行业温度、富途日线和账户快照。
- 不补发飞书；重生成流程不得调用真实通知器。
- Dashboard 在下一次刷新后读取新的基础文件。

这是一项明确限定到 `2026-07-15` 的一次性纠错操作，不新增通用“任意覆盖冻结
报告”产品能力。

## 兼容性与非目标

- 美股和港股候选、仓位、卖出与 Dashboard 报告语义保持不变。
- 不自动下单。
- 不实现波动率放大信号。
- 不增加新的依赖、主题、图标库或前端框架。
- 不修改其他券商账户卡和持仓表。
- JSON 新字段采用向后兼容的加法扩展，不提升 schema version。

## 错误处理

- 候选必需字段缺失：排除并记录原因。
- 行业 ID 缺失或行业温度请求缺行：候选排除，其他候选继续。
- 行业温度请求整体失败：报告生成失败，不使用缓存外的旧行业温度冒充当天数据。
- 富途候选日线单标的失败：该候选以 ATR 或执行价不可用排除。
- 富途系统性故障、账户缺失或报告文件原子替换失败：整份重生成失败，旧完整报告
  保持不变。
- 未知动作、温度、节气或原因代码：进入人工复核或排除，不能静默降级。

## 验证与验收

最小自动化覆盖：

- 12 条买入纪律的通过与边界：`priceIndex = 200`、强度 `= 95`、市值 `= 100`、
  日成交额 `= 2` 均通过。
- `热 → 4%`、`沸 → 2%`，以及 A 股 100 股取整。
- `谷雨 / 立夏 / 夏至` 通过，`小暑 / 大暑 / 立秋` 排除。
- 行业温度 `热 / 沸` 通过，`温 / 平 / 凉` 及缺失排除。
- 原 `右侧天数 < 10` 不再排除处于允许节气的候选。
- 保护线、危险、离开右侧、温度转平的卖出优先级。
- `沸腾 / 开香槟` 只上移保护线，不生成减仓。
- 已有持仓只显示失败提示，不因买入门槛失效卖出。
- 主表只包含正式动作，审计保留全部候选和排除原因。
- 两个价格均保存、标注且用途不混淆。
- 行业温度按去重行业 ID 批量请求。
- 一次性重生成不调用真实通知器，并原子覆盖完整报告对。

实际验证顺序：

1. 运行相关单元测试并记录精确结果。
2. 用真实 API 和富途行情覆盖重生成 `2026-07-15` 报告，核对 JSON、Markdown、
   进程版本和文件时间。
3. 确认没有新增飞书发送记录。
4. 检查并重启仍在运行旧代码的 Dashboard、launchd 或 screen 进程。
5. 运行 `make acceptance`。只有 `PASS` 才算完成。
6. 重新部署完全相同的已验收 Git SHA，验证新 PID、工作目录、Git SHA、新日志、
   HTTP 200 和桌面/移动浏览器流程，再提供评审 URL。
