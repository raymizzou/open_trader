# US Sentiment Change Monitor Design

## Goal

Add a US-only sentiment change monitor that helps the user notice meaningful
changes in external discussion and news around monitored US symbols.

The module is not a trading signal. It does not produce buy, sell, overweight,
underweight, target-price, stop-loss, or sizing recommendations. Its sole job is
to compare the latest TradingAgents sentiment/news reports against prior local
reports and surface changes that may deserve manual review.

## Scope

The first version covers only US symbols.

Included symbols:

- current US holdings from the dashboard portfolio
- US symbols present in the latest watchlist or trading plan artifacts
- US symbols present in the latest TradingAgents advice artifact when they are
  also part of one of the above monitored sets

Excluded symbols:

- HK symbols
- cash rows
- symbols with no latest TradingAgents advice
- symbols only found in old historical artifacts and not currently monitored

HK sentiment is intentionally out of scope because the current Yahoo Finance,
StockTwits, and Reddit sources often provide no useful HK discussion coverage.
HK rows should be skipped rather than reported as weak, bearish, or bullish.

## Source Data

The first version reuses reports already generated by TradingAgents and stored
inside `trading_advice.csv`.

For each advice row:

1. Parse `raw_decision` as JSON.
2. Read `state.sentiment_report`.
3. Read `state.news_report`.
4. Combine those two text blocks as the current source report for comparison.

This version does not independently fetch Yahoo Finance, StockTwits, Reddit,
or X data. It also does not store full raw post archives. The artifact schema
leaves room for future raw snapshot fields, but the first implementation should
not build a new crawler.

## Comparison Windows

Each monitored symbol is compared against two local baselines.

### Previous Report

Find the most recent earlier US `trading_advice.csv` row for the same symbol
before the current run date. This answers: what changed since the last report?

If no previous report exists, mark `previous_baseline.status` as
`insufficient_history`.

### Seven-Day Baseline

Find same-symbol US TradingAgents advice rows whose run dates fall in the seven
calendar days before the current run date. Use their sentiment/news report text
as the short-term normal baseline. This answers: is today's discussion unusual
relative to the recent week?

If fewer than two baseline reports exist in that window, mark
`seven_day_baseline.status` as `insufficient_history`.

## Output Artifacts

Dated artifacts:

```text
data/runs/<YYYY-MM-DD>/US/sentiment_changes.json
reports/sentiment_changes/<YYYY-MM-DD>-US.md
```

Latest artifact:

```text
data/latest/US/sentiment_changes.json
```

The module follows the existing Open Trader artifact rules:

- write dated artifacts first
- promote latest only when the surrounding US daily run promotes latest
- write JSON atomically
- never partially overwrite `latest` on failure
- keep missing data explicit instead of inventing a signal

## JSON Schema

Top-level artifact:

```json
{
  "schema_version": "open_trader.sentiment_changes.v1",
  "run_date": "2026-06-22",
  "market": "US",
  "source": "tradingagents_sentiment_and_news_reports",
  "records": []
}
```

Record shape:

```json
{
  "market": "US",
  "symbol": "VIXY",
  "status": "changed",
  "severity": "review",
  "current_source_hash": "sha256:...",
  "previous_baseline": {
    "status": "available",
    "run_date": "2026-06-19",
    "source_hash": "sha256:..."
  },
  "seven_day_baseline": {
    "status": "available",
    "start_date": "2026-06-15",
    "end_date": "2026-06-21",
    "samples": 4
  },
  "change_vs_previous": "StockTwits discussion shifted from mixed to more explicitly bearish around VIXY underperformance.",
  "change_vs_7d_baseline": "The current report emphasizes VIX suppression more than the recent baseline.",
  "new_topics": ["VIX reliability", "vix crush"],
  "intensified_topics": ["VIXY lagging"],
  "faded_topics": [],
  "risk_flags": ["discussion_change_requires_review"],
  "evidence": [
    {
      "source": "tradingagents_sentiment_report",
      "excerpt": "Retail traders expect a VIX crush and note VIXY lagging.",
      "source_run_date": "2026-06-22"
    }
  ],
  "decision_use": "manual_review_only",
  "error": ""
}
```

Allowed `status` values:

- `changed`
- `unchanged`
- `no_signal`
- `insufficient_baseline`
- `missing_source`
- `error`
- `skipped`

Allowed `severity` values:

- `none`
- `info`
- `review`
- `high_review`

`high_review` is for large or potentially material discussion changes, such as
a new regulatory theme, lawsuit theme, short-report theme, product incident, or
sharp shift from no discussion to high-engagement discussion. It still does not
mean buy or sell.

## Change Classification

Use a strict JSON LLM comparison step for the first implementation. The prompt
must instruct the model to compare source reports only and return the schema
above. It must prohibit trading recommendations.

The comparison prompt should ask for:

- new topics
- intensified topics
- faded topics
- changes in source quality or coverage
- changes in sentiment framing
- evidence excerpts
- whether the change deserves manual review

The comparison prompt must reject or downgrade:

- generic market commentary that is unchanged from the baseline
- unsupported claims not present in source text
- bullish or bearish labels used as trading direction
- source reports that mainly say no data was found

If the LLM returns invalid JSON, write a row with `status=error` and keep the
rest of the run usable.

## Dashboard Integration

Replace the current `新闻 / 舆论` placeholder card with a US-only
`舆论变化` card.

For US symbols, show:

- status label: `无变化`, `有变化`, `需复核`, `基线不足`, or `无有效信号`
- one-line change versus previous report
- one-line change versus seven-day baseline
- compact topic chips for new or intensified topics
- explicit footer text: `仅用于人工复核，不改变交易动作`

For HK symbols, do not show a misleading unavailable warning. The card should
either be hidden or show a quiet `暂不覆盖港股` state.

The dashboard must not translate `severity` into trade action labels. It should
not use green/red styling that implies buy/sell direction. Use neutral and amber
review states.

## CLI And Pipeline Integration

Add a manual CLI command:

```bash
.venv/bin/python -m open_trader detect-sentiment-changes \
  --advice data/latest/US/trading_advice.csv \
  --data-dir data \
  --reports-dir reports \
  --date 2026-06-22 \
  --market US \
  --update-latest
```

The US daily premarket runner should call the detector after TradingAgents advice
is written and before latest promotion. HK daily runs should skip the detector.

The detector should accept fake comparator implementations in tests so normal
test runs never call a real LLM.

## Error Handling

The module is conservative.

- Missing latest advice: fail the CLI with a clear error.
- Missing current `sentiment_report` and `news_report`: write
  `status=missing_source`.
- Source text says all sources are empty or unavailable: write
  `status=no_signal`.
- No previous or seven-day history: write `status=insufficient_baseline` unless
  a current source clearly contains a manually reviewable new event.
- Invalid LLM JSON: write `status=error` for that symbol and continue.
- Latest promotion failure: roll back to the previous latest artifact following
  the existing daily-run promotion pattern.

## Safety Boundaries

This module does not place orders, modify trading plans, modify trade actions,
or change action readiness.

It can only add manual-review context. Any future use in automation must pass a
separate design review.

The output must not contain provider credentials, API keys, or raw environment
values. Evidence excerpts should be short and traceable to the local
TradingAgents report text.

## Testing Strategy

Unit tests:

- load current US sentiment/news source text from fixture advice rows
- skip HK rows
- find previous same-symbol report
- build seven-day baseline from local dated artifacts
- mark insufficient history explicitly
- detect `no_signal` source reports
- write dated JSON and Markdown artifacts
- promote latest atomically
- continue when one symbol comparison fails

Comparator tests:

- fake comparator returns `changed`
- fake comparator returns `unchanged`
- invalid comparator JSON becomes `status=error`
- comparator output containing a trading recommendation is rejected

Dashboard tests:

- attach `sentiment_changes` to US holdings
- render `舆论变化` in place of the placeholder
- show `暂不覆盖港股` or hide the card for HK holdings
- show `仅用于人工复核，不改变交易动作`
- do not render buy/sell labels from sentiment severity

Pipeline tests:

- US daily run writes `sentiment_changes.json`
- HK daily run skips sentiment change detection
- latest promotion includes sentiment changes for US only
- latest rollback restores the previous sentiment changes artifact on failure

No automated test should call real TradingAgents, Yahoo Finance, StockTwits,
Reddit, X, or a live LLM.
