# 富途控制器连接生命周期设计

## 背景

2026-07-22，三个趋势市场控制器仍每 5 秒运行，但富途 OpenD 不再响应本地
协议握手。OpenD 进程和监听端口仍存在，上游连接也仍为已连接状态，因此页面
只看到“OpenD 不可达”，控制器则持续超时。

根因在控制器的连接生命周期：每轮都会为交易日判断、保护检查和账户读取创建
新的富途 context。三个控制器从 01:28 到 03:23 累计创建约 1.5 万个连接；正常
路径虽然调用了 `close()`，高频握手仍最终拖死 OpenD。之后富途 SDK 的同步连接
构造器无限重试，使控制器心跳冻结并继续形成连接风暴。

这是代码缺陷。两小时只让缺陷显现，不应把两小时等待加入 acceptance。

## 目标行为

- 保留每 5 秒一次的保护检查，不降低策略响应频率。
- 每个市场控制器复用一个行情连接，以及该市场需要的一个模拟账户读取连接。
- 日历判断、周期推导、保护检查和对账读取共用上述连接，不在每轮重新握手。
- 只有连接实际失败时才关闭并重建；控制器退出时关闭持有的连接。
- 真实下单连接仍只在存在待执行动作时临时创建，并沿用现有订单防重校验。
- OpenD 不可用时连接尝试必须有界失败，控制器继续更新阻塞状态和心跳，不能卡在
  SDK 构造器的无限重试中。
- 创建账户或下单 context 前先通过行情协议调用确认 OpenD 可用，避免监听端口仍在
  但协议已失效时进入同步连接死等。

## 最小实现

连接由 `run_trend_market_controller()` 的单个进程生命周期持有，不增加连接池、
后台守护进程或通用资源管理框架。

`_derive_cycle()` 和保护检查接受调用方提供的行情连接。调用方提供的连接属于
控制器，检查函数不得在成功返回时关闭它；函数自行创建的连接仍由函数自己关闭，
以保持命令行单次调用兼容。

富途模拟账户加载允许使用调用方提供的现有客户端。提供时只读取并转换账户快照，
不关闭客户端；未提供时保持当前创建、读取、关闭行为。实际下单继续使用独立的
短连接，不与只读账户连接共享。

行情客户端使用 SDK 已有的异步连接方式，并设置有限的同步查询连接超时。首次
协议查询成功后才允许继续创建账户或交易客户端。连接异常时，控制器关闭并清空
所持客户端，记录现有阻塞状态，下一轮按现有重试节奏重建。

## 测试与验收

- 用快速多循环测试运行多个 5 秒逻辑轮次，断言正常情况下行情和账户客户端各只
  创建一次；该测试在数秒内稳定复现旧代码的连接风暴，无需两小时 soak。
- 测试借入连接不被检查函数关闭，而函数自行创建的连接仍会关闭。
- 测试 OpenD 协议不可用时调用在限定时间内返回失败，控制器心跳可继续推进，并在
  下一轮重建连接。
- 测试只有实际待执行动作才创建下单客户端，现有防重测试继续通过。
- 停止旧控制器，重启 OpenD，再部署三个新控制器；检查 PID、工作目录、Git SHA、
  新日志、心跳和 OpenD 连接数在多个轮次中保持稳定。
- 直接刷新真实富途及老虎页面，确认行情和账户读取恢复。
- 最后只运行一次 `make acceptance`。只有 `PASS` 后才按同一 Git SHA 重启服务并
  提供 review URL。

## 不做

- 不用自动重启 OpenD 掩盖连接风暴。
- 不增加两小时 acceptance soak。
- 不引入连接池、跨进程共享连接或新的服务管理层。
- 不改变 5 秒策略频率、交易决策、订单防重规则或报告生成规则。
